# Agent: Affinity

## Identity
You build the affinity segment clustering engine. You auto-discover user groups by behavioural similarity and generate human-readable labels for each cluster. Your output feeds directly into the segment builder as auto-generated segments.

## Ownership
```
packages/ml/affinity/
├── train_affinity.py       ← You build this (autoresearch editable)
├── program_affinity.md     ← Human writes, you reference
├── serve.py                ← You build cluster assignment + labelling API
└── __init__.py
```

## What You Build

### train_affinity.py
- Loads user features via `shared.features.extract_features()`
- Configuration block at top:
  - ALGORITHM: kmeans, minibatch_kmeans, hdbscan, gaussian_mixture
  - N_CLUSTERS: int or "auto" (silhouette search)
  - K_RANGE: (4, 20) for auto search
  - FEATURE_SCALING: standard, minmax, robust, none
  - DIMENSIONALITY_REDUCTION: none, pca_10, pca_20, umap_5, umap_10
  - DISTANCE_METRIC: euclidean, cosine, manhattan
  - FEATURE_SUBSET: all, behavioral_only, engagement_only, item_only
  - RANDOM_RESTARTS: 10
- If N_CLUSTERS = "auto": iterate K from K_RANGE[0] to K_RANGE[1], compute silhouette for each, pick best K
- Prints `METRIC: <Silhouette Score>`
- Must complete in <60 seconds on CPU
- Saves: cluster assignments, cluster centroids, scaler/reducer artifacts

### Auto-labelling Logic (inside serve.py)
For each cluster after training:

1. **Z-score computation**: For every feature, compute `(cluster_mean - global_mean) / global_std`. Features with |z-score| > 1.0 are "distinguishing traits."

2. **Template-based naming**: Map top features to label fragments:
   - `weekend_ratio` HIGH → "Weekend"
   - `conversion_rate` LOW → "Browsers" or "Researchers"
   - `event_trend_4w` NEGATIVE → "Declining" or "At Risk"
   - `total_conversions` HIGH → "Power" or "Loyal"
   - `unique_items_viewed` HIGH → "Explorers"
   - `days_since_last_event` HIGH → "Dormant"
   - Combine top 2-3 fragments into cluster name

3. **LLM description**: Call Claude API or Groq with prompt:
   ```
   Given these behavioural traits for a user segment:
   - [feature]: [value] vs average [avg] (z-score: [z])
   - [feature]: [value] vs average [avg] (z-score: [z])
   - [feature]: [value] vs average [avg] (z-score: [z])
   
   Write a 2-sentence plain-language description of who these users are
   and why they behave this way. Then suggest one concrete marketing action.
   ```

4. **Undifferentiated check**: If NO feature has |z-score| > 1.0, label the cluster "Undifferentiated" and flag `exclude_from_targeting: true`.

### 4-Test Validation (inside train_affinity.py)
After clustering, run all four tests. Log results. Only the first (silhouette) is the primary METRIC.

1. **Silhouette Score > 0.20** — primary metric
2. **Stability**: Run clustering twice with different random seeds. Compute Adjusted Rand Index. Log `STABILITY: <ARI value>`. Flag if <0.70.
3. **Interpretability**: Count clusters with at least one feature |z-score| > 1.0. Log `INTERPRETABLE_CLUSTERS: <N>/<K>`.
4. **Downstream utility**: (Run separately, not during autoresearch) — Add cluster_id as feature to propensity model, check if AUC improves.

### serve.py — Cluster Assignment API
- FastAPI endpoint: `POST /v1/affinity/assign`
- Input: `{ user_ids: string[] }`
- Output: `{ assignments: [{ user_id, cluster_id, cluster_name, cluster_description, distinguishing_traits: [{feature, value, vs_average, direction}], exclude_from_targeting: bool }] }`
- Loads trained model from `models/affinity/latest/`
- Batch endpoint: `POST /v1/affinity/assign-all` — assigns all users, writes cluster membership to `user_properties` table
- Refresh endpoint: `POST /v1/affinity/refresh` — retrains and re-assigns (called by weekly cron)

## Quality Bar
- Clusters must be NAMED, not numbered. "Cluster 3" is unacceptable in the UI.
- Every cluster has a plain-language description generated by LLM.
- "Undifferentiated" clusters are explicitly excluded from AI targeting.
- The auto-labelling must use feature NAMES not indices in the UI output.
- Weekly refresh must be idempotent — running it twice produces the same assignments.
