## Critical context (read before touching anything)

1. **The connector emits `order_placed`, not `order_completed`.** The ecommerce pack defaults to `order_completed`; we manually updated the two affected GWM goals via `UPDATE prediction_goals SET target_event = 'order_placed' WHERE …`. Same UPDATE will be needed for any new connector-driven tenant until the pack is fixed.

2. **ML service has two `.env` files and PM2 caches env on first start.**
   - `/var/www/html/storees-ml/.env` (root)
   - `/var/www/html/storees-ml/packages/ml/.env` (inner — what `python-dotenv` actually loads since PM2 launches uvicorn from `packages/ml/`)
   - PM2 process is `id 1` named `storees-ml`. After editing `.env`, restart with **`pm2 restart 1 --update-env`** — plain `pm2 restart 1` keeps cached env.
   - `MIN_POSITIVE_LABELS=50` set (vs code default 200), which is what made dormancy/churn trainable.

3. **The 50K-row score-count drift will self-heal on next backend deploy** when migration 0058 runs. After deploy: open Repeat Purchase Propensity detail page — "Total Scored" should drop from ~20K to ~13–15K (distinct customers).

4. **`MlTrainResult.brier_before_calibration` / `_after`** are returning in the API payload but **not surfaced in the UI yet** — see open item #3.

5. **Live-eval rows exist in the DB but are not rendered** — the training-history endpoint returns them (status='live_eval'), the sparkline filters them out (only status='success'). See open item #4.

## Open items (prioritised)

### P1 — Cart Abandonment fix

Goal still returns `INSUFFICIENT_DATA: 0 positive labels`. Root cause: `_label_cart_without_checkout` in [packages/ml/shared/prepare.py](packages/ml/shared/prepare.py) needs `checkout_completed` events which the connector doesn't emit. Two options:

- **Cheap fix:** change the label function to treat `order_placed` as the "did checkout" signal. ~10-line Python change.
- **Cleaner fix:** the connector emits a synthetic `checkout_completed` whenever an order row arrives. ~20 lines in `dataSyncService.ts`.

### P2 — Pack default fix (`order_completed` → flexible)

The ecommerce pack at [packages/backend/src/packs/ecommerce.json](packages/backend/src/packs/ecommerce.json) hardcodes `order_completed`. Every connector-driven onboarding will hit the same trap. Options:

- Pack accepts a list of target events and tries each (`["order_completed", "order_placed"]`)
- Or onboarding code introspects which event names actually exist in the project's `events` table and picks accordingly

### P3 — Surface calibration improvement in UI

Backend already returns `brier_before_calibration` / `brier_after_calibration` on training results. UI doesn't show them. Easy add to the Predictions card or goal-detail page.

### P4 — Overlay live-AUC on the sparkline

The `TrainingTrend` sparkline in [page.tsx](packages/frontend/src/app/(dashboard)/analytics/predictions/page.tsx) currently filters to `status === 'success'`. The live-eval data (`status === 'live_eval'`) is sitting in the same table. Overlay it as a second line so you can see train-AUC vs live-AUC over time — the *real* drift signal.

### P5 — Auto-promote / shadow-score (the missing half of champion-challenger)

Today the version registry tracks every train and lets you manually promote/rollback. The full shadow-score automation (train challenger → score in parallel for N days → compare live AUC → auto-promote) is the next layer. Substrate is all in place: `prediction_model_versions` table, `/promote` endpoint, live-eval scheduler. Need a comparison job + decision logic.

### P6 — Consolidate ML service env

Pick one `.env` location (suggest **`/var/www/html/storees-ml/packages/ml/.env`** since that's where python-dotenv loads from) and delete the other. Also add a `pm2 ecosystem.config.js` so future env changes don't need `--update-env`. Lower priority but stops the next 2-hour debug.

### P7 — Read-only DB role for ML service

Security smell. The ML service only reads `customers`, `events`, `agents` but currently connects as `virpanai_user` which has write access. Create a `virpanai_ml_ro` role with `GRANT SELECT` only.
